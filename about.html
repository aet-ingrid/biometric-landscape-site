Framework Primer

Read this once before engaging with the comparison data, or return to it when a term is unclear. Covers the layer model, binding quality taxonomy, key concepts, and how to read a player brief.

01 What is biometric anchoring?

Biometric anchoring is the cryptographic linkage of a biometric proof event to a credential artifact that travels with the identity claim. A biometrically-anchored credential carries within itself — or in a cryptographically-linked signed record — evidence that a specific biometric event occurred at issuance or at the moment of consequence. A relying party receiving the credential can verify that evidence independently, without querying a separate audit system or trusting the issuer's word after the fact.

This distinguishes biometric anchoring from conventional password-based or assertion-based identity models. A password-based system proves knowledge of a secret; an assertion-based system proves the issuer vouched for an identity at some point. Neither model creates a cryptographic chain from a physical fact about a specific human body to the artifact a downstream system accepts. Biometric anchoring does. The policy consequence: under regulatory frameworks that require artifact-level non-repudiation — such as EU AI Act Article 14 human-in-the-loop requirements for high-risk AI — the difference between anchored and unanchored credentials is not cosmetic. It determines whether an audit can be resolved from the credential artifact alone or whether it requires the issuer's cooperation to reconstruct.

02 The layer model

The research uses a three-layer model to place each player's role in the identity stack. A player may occupy one, two, or all three layers.

Layer 2 — top of stack
Binding Authority / IAM Orchestrator
The actor that receives Layer 1 results and issues the downstream credential — the session token, OAuth JWT, W3C VC, or agent identity — that travels to relying parties. This is the policy and orchestration layer.
Example actors from this research: Microsoft Entra ID; Okta; Ping Identity (PingOne, PingFederate)
↑ verification results and signed credentials flow up from Layer 1
Layer 1
Biometric Verification / IDV Supplier / Credential Issuer
The actor that captures a biometric, verifies liveness, matches the biometric to a claimed identity, and produces a verification result or signed credential. Converts a physical event into a digital claim.
Example actors from this research: World / Tools for Humanity (Orb + ZK credential issuance); Keyless sMPC (Ping Identity, acq. Jan 2026); iProov (Okta partner); Nametag (Okta partner)
↑ hardware-attested biometric events flow up from Layer 0
Layer 0 — hardware foundation
Hardware Substrate
The physical root of trust: the chip, secure enclave, or sensor that hardware-enforces liveness or key storage. No software can impersonate hardware physics at this layer.
Example actors: Intel PTT / AMD fTPM / Infineon (TPM 2.0); Apple Secure Enclave; Android StrongBox; World Orb (iris hardware); Thales Luna HSM

Why layer position matters for binding quality: A Layer 2 orchestrator that does not own Layer 1 cannot guarantee the biometric proof enters the artifact it issues — it can only condition token issuance on a result it receives from a supplier. A player that owns both Layer 1 and Layer 2 has native control of the full chain from biometric event to signed credential. Ping Identity is the only enterprise IAM orchestrator in this research that owns both layers following the Keyless acquisition.

03 Binding quality explained

Binding quality is the primary analytical dimension in this research. Three values are used:

Hard
hard

Cryptographic chain from biometric proof to artifact; independently verifiable. The biometric event is not merely recorded — its proof is embedded in or cryptographically linked to the artifact a relying party accepts. A verifier can confirm the biometric proof without querying the issuer.

cryptographic chain independently verifiable no issuer query needed
In practice: Windows Hello for Business — the biometric gesture unlocks a TPM key that signs the authentication assertion. The Entra-issued PRT is bound to the TPM session key. The AIK certificate lets any verifier confirm the key is hardware-bound. World ID — a ZK-SNARK proof of Merkle tree membership is verifiable on-chain without querying Tools for Humanity.
Hard Path Native
hard path native

Player owns the cryptographic mechanism; integration into the artifact signing chain not confirmed shipped. The player controls the technology that would produce hard binding — a biometric-derived key, a ZK proof system — but public documentation does not confirm that mechanism has been wired into the issued credential artifact.

mechanism owned integration pending not yet confirmed shipped
In practice: Ping Identity Universal Services — Keyless sMPC produces a biometric-derived cryptographic key (not just accept/reject), which is the native mechanism for hard binding. Whether that key is currently embedded in the artifact signing chain is not publicly confirmed. Entra Verified ID Face Check — both artifacts are signed and the FaceCheck receipt JWT binds the comparison outcome, but the face match is probabilistic rather than cryptographic identity equivalence.
Soft
soft

Audit log or policy signal only; no cryptographic proof on the artifact. The biometric event happened — and was recorded — but the credential artifact a relying party receives contains no cryptographic reference to it. Confirming the biometric underlies a specific token or agent action requires querying a separate audit system.

audit-log binding only issuer query required no biometric proof on artifact
In practice: Entra Agent ID / Okta Agent Identity — an AI agent gets an OAuth service principal. Even if the provisioning administrator was authenticated via Windows Hello (a hard-binding event for that person), the biometric proof does not transfer to the agent credential. The agent's OIDC JWT carries no cryptographic reference to any biometric event.

The agentic distinction: These three values apply to AI agent credentials as well as human credentials. A biometric-provisioned agent governance model (type b) — where the human administrator was biometrically verified at provisioning but the proof does not transfer — is still soft binding. A biometric-anchored agent credential (type a) — where the biometric proof is cryptographically embedded in the agent identity record — would be hard binding. No player in this research has shipped a type (a) agent credential as of April 2026.

04 Key concepts glossary
Layer 0
Hardware substrate: the chip, HSM, or purpose-built sensor that provides a hardware root of trust. Examples: TPM 2.0, Apple Secure Enclave, Thales Luna HSM, World Orb.
Layer 1
IDV / biometric verification supplier / credential issuer. The actor that converts a physical biometric event into a digital verification result or signed credential.
Layer 2
Binding authority / IAM orchestrator. The actor that receives Layer 1 results and issues the downstream credential — token, JWT, W3C VC — that relying parties accept.
Hard binding
A cryptographic chain from biometric proof to credential artifact. Independently verifiable without querying the issuer. Supports non-repudiation.
Soft binding
Audit log or policy signal only. The biometric event was recorded; the artifact contains no cryptographic proof of it. Confirming the connection requires querying the issuer's records.
Hard path native
The player owns the cryptographic mechanism that enables hard binding (e.g., a biometric-derived key, a ZK proof system), but integration into the artifact signing chain is not yet publicly confirmed as shipped.
PoP — Proof of Personhood
A credential that attests a human is unique and real, without necessarily revealing legal identity. World ID's ZK-SNARK credential is the primary PoP example in this research.
IAM — Identity and Access Management
Systems for managing digital identities and controlling access to resources. Enterprise IAM orchestrators (Microsoft Entra, Okta, Ping Identity) are Layer 2 actors in this model.
ZK — Zero-Knowledge Proof
A cryptographic proof that demonstrates knowledge of a fact without revealing the underlying data. World ID uses ZK-SNARKs (Groth16, Semaphore protocol) to prove Merkle tree membership without revealing which identity committed to which leaf.
VC — Verifiable Credential
A W3C-standard tamper-evident credential with a cryptographic proof. The issuer signs the VC; the holder presents it; the verifier confirms the issuer's signature without querying the issuer. Used by Microsoft (Entra Verified ID) and Ping Identity (PingOne Neo).
Agentic
Covers AI agent identity — not only human identity. An agentic lane issues credentials to AI agents operating on behalf of humans. Binding quality determines whether those agent credentials carry cryptographic proof of human authorization or only governance-log evidence.
Agentic distinction: type (a) vs type (b)
Type (a) — biometric-anchored agent credential: biometric proof is cryptographically embedded in the agent identity record at provisioning; independently verifiable at point of consequence. Type (b) — biometric-provisioned agent governance: the provisioning administrator may have been biometrically verified, but the proof does not transfer to the agent credential. All four players in this research are currently type (b) for agentic lanes.
sMPC — Secure Multi-Party Computation
A cryptographic protocol where computation is distributed across multiple parties with no single party holding all inputs. Keyless (acquired by Ping Identity, Jan 2026) uses sMPC to produce a biometric-derived cryptographic key without any node storing the raw biometric.
MCP — Model Context Protocol
Anthropic's open protocol for AI tool and resource access. Used by Microsoft and Ping Identity for agent authorization enforcement at the tool/resource level.
XAA — Cross App Access
Okta's proprietary open protocol (published June 2025) extending OAuth via the Identity Assertion Authorization Grant to shift agent-to-app access control to Okta as central identity provider.
WIMSE — Workload Identity in Multi-System Environments
IETF working group defining standards for AI agent workload identity across system boundaries. Ping Identity (Brian Campbell) is a co-author of the core WIMSE specification.
05 How to read a player brief

Each player brief is organized as six sections. The lane card is the primary data element — here is an annotated example using a simplified version of the actual component:

Example Lane — Biometric Credential

Binding quality badge The green/amber/slate badge in the top-right corner. This is the primary classification: hard, hard path native, or soft. Read the note below it to understand why.
Lane ID + role The small text below the lane name: player ID, layer (1 or 2), issuing authority context, and whether the lane covers agentic (AI agent) identity.
Binding quality note The paragraph under the header. One sentence of authoritative explanation — do not paraphrase this when citing the research.
Artifact type pill The blue pill (w3c_vc, fido2_assertion, oidc_jwt, zk_credential) — what gets issued. This determines what a relying party receives and what it can independently verify.
Coverage pills Bottom row: which binding type dimensions this lane covers (PoP, agent, action, credential, device) and at what quality level. A lane may cover multiple dimensions at different quality levels.

The six sections of each player brief:

  • 01 Key Finding — the most important analytical claim about this player; the one thing to carry forward
  • 02 Lanes — each lane card with binding quality, artifact type, agentic flag, and coverage
  • 03 Architecture — how the binding chain works technically; why each binding quality value was assigned
  • 04 Strategic Position — entity type, revenue model, scale, competitive context
  • 05 Gaps & Regulatory Exposure — critical gap, regulatory forcing functions, protocol adoption table
  • 06 Sources — all primary and secondary sources, verified via live web research April 2026
